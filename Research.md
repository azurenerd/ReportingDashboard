``` **Date-to-X mapping:** Linear interpolation from the date range (Jan 1 – Jun 30) to SVG width (0 – 1560px). Implement as a simple helper method in the `@code` block:
```csharp
private double DateToX(DateOnly date)
{
    var start = new DateOnly(2026, 1, 1);
    var end = new DateOnly(2026, 6, 30);
    var totalDays = end.DayNumber - start.DayNumber;
    var elapsed = date.DayNumber - start.DayNumber;
    return (elapsed / (double)totalDays) * 1560;
}

## 4. Architecture Recommendations

### Pattern: Single-Page, Service-Injected, Flat Data Model

This is not a CRUD app, not a SPA, and not a microservice. It is a **read-only data renderer**. The architecture should be as flat as possible:

```
data.json → DashboardDataService (singleton) → Dashboard.razor → HTML/CSS/SVG
```

No repository pattern. No CQRS. No MediatR. No layers of abstraction. The entire data pipeline is:

1. `Program.cs` registers `DashboardDataService` as a singleton.
2. `DashboardDataService.LoadAsync()` reads `wwwroot/data.json`, deserializes to `DashboardData`, caches in memory.
3. `Dashboard.razor` injects `DashboardDataService`, calls `GetData()`, renders.

### SVG Timeline Rendering Strategy

The timeline is the most complex visual element. Render it as inline SVG within Blazor:

```razor
<svg width="1560" height="185" style="overflow:visible">
    @* Month grid lines *@
    @foreach (var (month, x) in monthPositions)
    {
        <line x1="@x" y1="0" x2="@x" y2="185" stroke="#bbb" stroke-opacity="0.4" />
        <text x="@(x+5)" y="14" fill="#666" font-size="11">@month</text>
    }

    @* "NOW" line *@
    <line x1="@nowX" y1="0" x2="@nowX" y2="185"
          stroke="#EA4335" stroke-width="2" stroke-dasharray="5,3" />

    @* Milestone tracks *@
    @foreach (var (track, y) in trackPositions)
    {
        <line x1="0" y1="@y" x2="1560" y2="@y"
              stroke="@track.Color" stroke-width="3" />

        @foreach (var evt in track.Events)
        {
            @if (evt.Type == "poc")
            {
                <polygon points="@DiamondPoints(DateToX(evt.Date), y)"
                         fill="#F4B400" filter="url(#sh)" />
            }
            @* ... other event types *@
        }
    }
</svg>
```

**Date-to-X mapping:** Linear interpolation from the date range (Jan 1 – Jun 30) to SVG width (0 – 1560px). Implement as a simple helper method in the `@code` block:

```csharp
private double DateToX(DateOnly date)
{
    var start = new DateOnly(2026, 1, 1);
    var end = new DateOnly(2026, 6, 30);
    var totalDays = end.DayNumber - start.DayNumber;
    var elapsed = date.DayNumber - start.DayNumber;
    return (elapsed / (double)totalDays) * 1560;
}
```

## 6. Risks & Trade-offs

### Risk: Blazor Server Overhead for a Static Page

**Concern:** Blazor Server maintains a SignalR WebSocket connection and server-side component tree. For a read-only screenshot page, this is architecturally heavier than static HTML.

**Mitigation:** The overhead is negligible for a single local user. Blazor Server gives us hot reload, C# data binding, and a proper project structure. The alternative (generating static HTML from a console app) loses the live-preview workflow that makes `data.json` editing fast.

**Trade-off accepted:** Slight resource overhead in exchange for excellent developer experience.

### Risk: SVG Layout Drift from Design Reference

**Concern:** Hand-coding SVG positions in Razor could drift from the original `OriginalDesignConcept.html` pixel positions.

**Mitigation:** Extract exact coordinates from the reference HTML. Use the same `viewBox`, `width`, `height`, and position values. Overlay-compare screenshots during development.

### Risk: data.json Schema Changes Breaking the Page

**Concern:** If someone edits `data.json` with missing fields or wrong types, the page crashes with a deserialization error.

**Mitigation:** Add null-coalescing defaults in the model and a try-catch in the data service that shows a friendly error banner instead of a crash. Use nullable reference types and `[JsonIgnore]` defaults.

```csharp
public record HeatmapRow(
    Dictionary<string, string[]> Items
)
{
    public string[] GetItems(string month) =>
        Items.TryGetValue(month, out var list) ? list : Array.Empty<string>();
}
```

### Risk: Fixed 1920×1080 Layout Doesn't Fit Other Screens

**Concern:** The design is hardcoded to 1920×1080. On a laptop screen, content is clipped.

**Mitigation:** This is intentional — the output is for screenshots, not interactive use. Add a note in the README: "Open in a browser window at 1920×1080 or use browser zoom to fit." Alternatively, use `transform: scale()` CSS to auto-fit the viewport:

```css
@media (max-width: 1920px) {
    body { transform: scale(calc(100vw / 1920)); transform-origin: top left; }
}
```

### Non-Risk: "Why Not Blazor WebAssembly?"

Blazor WASM would eliminate the SignalR connection but adds: a 10+ MB download, slower startup, and no file-system access (would need to embed `data.json` differently). For a local tool, Blazor Server is strictly superior.

## 7. Open Questions

1. **Date range flexibility:** Should `data.json` define the timeline date range (e.g., Jan–Jun), or should it always auto-compute from the milestone dates? The design shows 6 months. If projects span 12+ months, the SVG needs to scale.

2. **Number of months in heatmap:** The design shows 4 months (Jan–Apr). Should this be configurable in `data.json` (e.g., show last 6 months)? This affects the CSS Grid column definition.

3. **Multiple projects:** Will this ever render more than one project per page? If so, the `data.json` schema and page layout would change significantly. Decide now.

4. **Automated screenshot capture:** Will screenshots be taken manually (browser → Print Screen / Snipping Tool) or automated (e.g., Playwright)? If automated, we should ensure the page loads all data before the screenshot is captured (add a "ready" signal).

5. **Data source for real use:** Will `data.json` be hand-edited, or generated from ADO/Jira via a script? If generated, should the Blazor app also include the generation logic, or is that a separate tool?

6. **Branding variations:** The design uses Segoe UI and Microsoft-esque colors. Will different teams want different color schemes? If so, we should support theme overrides in `data.json`.
